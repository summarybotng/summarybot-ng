# ADR-032: Email Content Templates and Landing Pages

## Status
Proposed

## Context

ADR-030 introduced email delivery for summaries, but the current implementation has limitations:

1. **Fixed template**: Email content is hardcoded with no customization options
2. **No source attribution**: Recipients don't know which platform/channel the summary came from
3. **No deep links**: Recipients cannot view the full summary with all details online
4. **No guild defaults**: Admins must manually enter email addresses each time
5. **All-or-nothing access**: Sharing requires full authentication or no authentication

Users need:
- Customizable email templates per guild
- Clear source attribution (Discord channel, WhatsApp group, etc.)
- Shareable links to full summary details
- Configurable access control for shared links
- Guild-level default recipients for automated summaries

## Decision

### 1. Email Template System

#### 1.1 Template Structure

Templates use Jinja2 with a hierarchical override system:

```
Priority (highest to lowest):
1. Per-summary custom template (API parameter)
2. Guild-specific template (database)
3. Server default template (file system)
4. Built-in fallback template (code)
```

#### 1.2 Available Template Variables

All summary fields are available as template variables:


#### 2.4 Security Model

**Token isolation**: Each token grants access to exactly one summary. Having a token for summary A does not grant any access to summary B or any other guild resources.

**Token generation**:
```python
import secrets
token = secrets.token_urlsafe(32)  # 256-bit entropy
```

**Access validation**:
```python
async def validate_access_token(token: str) -> SummaryAccessToken:
    token_record = await token_repository.get(token)

    if not token_record:
        raise HTTPException(404, "Invalid or expired link")

    if token_record.expires_at and token_record.expires_at < datetime.utcnow():
        raise HTTPException(410, "This link has expired")

    if token_record.max_access_count and token_record.access_count >= token_record.max_access_count:
        raise HTTPException(410, "This link has reached its access limit")

    return token_record
```

### 5. Interactive Email Editing

#### 5.1 Pre-Send Preview & Edit Flow

Users can review and modify email content before sending. This enables:
- Editing subject line and body text
- Adding or removing recipients
- Reviewing rendered content in context
- Making one-time changes without affecting templates

**UI Flow:**
```
Click "Send Email"
→ Preview modal opens with rendered email
→ User can edit subject, body, recipients
→ Click "Send" to deliver
```

#### 5.4 Edit Behavior

**Subject editing:**
- Direct text input
- Character limit enforced (150 chars)
- Cannot be empty

**Body editing:**
- Rich text or Markdown editor
- Preserves template structure if minor edits
- Full override if substantial changes
- Changes are per-send only (do not affect templates)

**Recipient editing:**
- Add/remove individual addresses
- Validate email format
- Cannot send with empty recipient list

#### 5.5 Implementation Notes

1. **Preview generates landing page token**: The preview endpoint creates the access token so the landing page URL is accurate in the preview. If the user cancels, the token is cleaned up after a short TTL.

2. **Markdown to HTML conversion**: User edits in Markdown are converted to HTML matching the template styles before sending.

3. **Original template preserved**: The `template_variables` field in the response allows the UI to show what data was available, but edits replace the rendered output.

4. **Audit trail**: When `subject_override` or `body_override` is used, the email delivery record logs that user modifications were applied.

## Security Considerations

1. **Token entropy**: Use `secrets.token_urlsafe(32)` for 256-bit tokens
2. **Token isolation**: Tokens grant access to exactly one resource
3. **Expiry enforcement**: Check expiry on every access
4. **Rate limiting**: Limit token validation attempts per IP
5. **Audit logging**: Log all token creation and access
6. **No enumeration**: Invalid tokens return same error as expired
7. **HTTPS only**: Landing pages must use HTTPS
8. **CSP headers**: Prevent embedding in iframes

## Consequences

### Positive
- Flexible email templates per guild
- Clear source attribution for recipients
- Secure, granular sharing via landing pages
- Reduced friction with guild default recipients
- Better admin control over email delivery

### Negative
- Increased database storage for tokens and settings
- More complex email sending flow
- Template customization requires careful validation
- Landing pages need separate styling/maintenance
